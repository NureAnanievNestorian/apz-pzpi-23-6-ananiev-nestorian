2 ЗАВДАННЯ
     Метою роботи є дослідження архітектури складної програмної системи та аналіз застосованих архітектурних підходів.
     У межах роботи необхідно виконати аналіз архітектури програмної системи, визначити архітектурний стиль, описати основні компоненти, проаналізувати взаємодію підсистем, оцінити масштабованість та переваги архітектурних рішень.

3 ОПИС ВИКОНАНОЇ РОБОТИ
3.1 Загальні відомості про систему
     Назва системи – Discord. Розробник – Discord Inc., компанія заснована у 2015 році в Сан-Франциско, США. Discord є розподіленою комунікаційною платформою реального часу, що забезпечує текстовий чат, голосовий та відеозв’язок, обмін файлами. Платформа підтримує шість клієнтських середовищ: Windows, macOS, Linux, iOS, Android та веб-браузер. Станом на 2024 рік сервіс обслуговує понад 200 мільйонів щомісячно активних користувачів, обробляє мільярди повідомлень щодня та підтримує понад 12 мільйонів одночасних WebSocket-підключень, що робить його показовим об’єктом для архітектурного аналізу.
3.2 Архітектурний стиль системи
     Discord використовує гібридну архітектуру, що поєднує два стилі. Перший – моноліт у поєднанні з мікросервісами: центральний монолітний REST API на Python обслуговує усталену бізнес-логіку, зокрема управління акаунтами, серверами, каналами та дозволами, тоді як близько двадцяти мікросервісів на Elixir/BEAM реалізують комунікацію реального часу, а окремий шар на Rust взаємодіє безпосередньо з базою повідомлень. Другий стиль – подієво-орієнтована архітектура: усі події системи, а саме нові повідомлення, зміни статусу та реакції, передаються через внутрішню шину pub/sub, що розв’язує компоненти між собою та забезпечує повністю асинхронну обробку.
     Такий вибір архітектури обґрунтований технічними вимогами: монолітний підхід зберігає стабільність і простоту розробки для бізнес-логіки, що рідко змінюється, а мікросервіси на BEAM-платформі забезпечують горизонтальне масштабування й ізоляцію збоїв для підсистем реального часу з мільйонами одночасних з’єднань.
3.3 Основні компоненти системи
     Архітектура системи охоплює чотири рівні. Клієнтський рівень представлений веб-застосунком на React, десктопним клієнтом на Electron з React та мобільними застосунками на React Native для iOS і Android. Edge-рівень включає Cloudflare, що забезпечує CDN, захист від DDoS та DNS-балансування навантаження, а також GCP L7 Load Balancer для розподілу вхідного трафіку між сервісами.
     Сервісний рівень складається з монолітного REST API на Python для обробки HTTP-запитів і бізнес-логіки, сервісу Gateway на Elixir/BEAM для управління постійними WebSocket-з’єднаннями із шардингом за guild_id, сервісу Presence на Elixir для відстеження онлайн-статусу, сервісу Voice/Video на WebRTC/UDP для голосового та відеозв’язку, Rust Data Services для хешування запитів і захисту від «гарячих» партицій, а також шини подій Event Bus за моделлю pub/sub. Рівень зберігання даних представлений ScyllaDB (72 вузли, написана на C++ без JVM) для зберігання повідомлень через CQL, PostgreSQL для метаданих серверів і каналів, Redis для кешування та даних присутності, а також BigQuery для аналітики. Уся інфраструктура розгорнута на Google Cloud Platform у Kubernetes.
3.4 Взаємодія компонентів
     Взаємодія між компонентами системи відбувається за двома основними протоколами: HTTPS для разових REST-запитів і WebSocket для постійних з’єднань реального часу. Внутрішня комунікація між сервісами реалізована через шину подій pub/sub.
     Сценарій відправки повідомлення охоплює сім кроків. Клієнт надсилає запит POST /channels/{id}/messages до REST API. REST API валідує токен і передає запит до Rust Data Service. Rust Data Service виконує вставку запису у ScyllaDB. ScyllaDB повертає підтвердження. Rust Data Service публікує подію MESSAGE_CREATE через Event Bus до Gateway. REST API асинхронно повертає клієнту відповідь 200 OK, не очікуючи завершення розсилки. Gateway виконує WebSocket push до всіх учасників каналу за механізмом fan-out. Асинхронність кроків 6 і 7 є ключовою: HTTP-відповідь і реалтайм-розсилка не блокують одна одну.
     Шардинг Gateway здійснюється за формулою shard_id = (guild_id » 22) % num_shards. Зсув ідентифікатора гільдії на 22 біти забезпечує рівномірний розподіл навантаження та горизонтальне масштабування без втрати локальності даних.
3.5 Масштабованість та надійність
     Балансування навантаження реалізоване на двох рівнях: Cloudflare і GCP L7 Load Balancer розподіляють трафік на рівні DNS та HTTP, тоді як Gateway шардується по guild_id, що дозволяє лінійно нарощувати кількість вузлів зі зростанням аудиторії. Відмовостійкість забезпечується моделлю акторів BEAM (Erlang VM), яка ізолює збої на рівні окремих процесів; supervision trees автоматично перезапускають збійні сервіси без зупинки системи в цілому. Redis використовується для кешування метаданих і даних присутності, що суттєво знижує навантаження на основні бази даних. ScyllaDB забезпечує реплікацію даних між вузлами з конфігурацією зберігання RAID 0+1 на базі NVMe та Persistent Disk для максимальної пропускної здатності. Kubernetes на GCP забезпечує автоматичне масштабування, оркестрацію контейнерів та відновлення після збоїв.
     Еволюція бази даних повідомлень наочно ілюструє підхід Discord до масштабування. У 2015 році використовувався один MongoDB-кластер, який не вміщував індекси в пам’яті вже при 100 мільйонах повідомлень. У 2017 році відбулася міграція на Apache Cassandra з дванадцятьма вузлами. До 2022 року кількість вузлів зросла до 477, проте з’явилися проблеми «гарячих» партицій і затримок через JVM GC з показником p99 на рівні 40–125 мс. Міграція на ScyllaDB дозволила скоротити парк до 72 вузлів при тих самих обсягах даних, знизивши p99-затримку читання до стабільних 15 мс – у 6,6 рази менше вузлів при кращих показниках продуктивності.
3.6 Приклади програмної реалізації
     Демонстрація роботи з Discord Gateway охоплює три етапи підключення. На першому етапі (Handshake) клієнт отримує пакет Hello від Gateway та зчитує значення heartbeat_interval для підтримки з’єднання активним. На другому етапі (Ідентифікація) клієнт надсилає пакет Identify з opcode 2, токеном автентифікації та бітовою маскою інтентів – фільтрів подій, на які клієнт підписується. На третьому етапі (Життєвий цикл) паралельно запускається фонова задача Heartbeat і розпочинається обробка вхідного потоку подій у реальному часі. Фрагменти програмного коду, що ілюструють усі три етапи, наведено у Додатку В. Слайди та відеодоповідь створені відповідно до встановлених вимог дисципліни. Презентація містить десять слайдів: титульний, предмет дослідження, структура доповіді, архітектурний стиль, діаграма компонентів, діаграма послідовності відправки повідомлення, демонстрація Gateway Client, еволюція бази даних, висновки та список джерел. Слайди наведено у Додатку Б, відеозапис доповіді – у Додатку А.


4 ВИСНОВКИ
     У результаті виконання практичного заняття досліджено архітектуру розподіленої комунікаційної платформи Discord. Визначено її архітектурний стиль – гібридна архітектура, що поєднує монолітний REST API на Python із мікросервісами на Elixir/BEAM і подієво-орієнтованою шиною pub/sub. Проаналізовано взаємодію компонентів на прикладі сценарію відправки повідомлення з механізмом fan-out через WebSocket. Оцінено переваги ключових архітектурних рішень: шардинг Gateway за guild_id забезпечує горизонтальне масштабування без втрати локальності; модель акторів BEAM ізолює збої на рівні процесів і автоматично відновлює сервіси; послідовна еволюція сховища MongoDB – Cassandra – ScyllaDB демонструє принцип «еволюція замість перепроєктування» і дозволила скоротити парк з 477 до 72 вузлів зі зниженням p99-затримки з 125 до 15 мс. Сформовано практичні навички архітектурного аналізу програмного забезпечення реального масштабу.

Запити до ШІ та приклади програмного коду
Інструмент: Claude (Anthropic), модель claude sonnet 4.6.
     Мета використання: формування базових чернеток для опису архітектури системи, уточнення технічних деталей та генерація прикладу програмного коду для демонстрації взаємодії з Gateway API.
     Внесок автора: усі відповіді перевірені та адаптовані автором; до звіту включено лише верифіковані факти.
     Запит В.1 до Claude (Anthropic). Тема: загальна характеристика архітектури Discord.
     Опиши архітектуру Discord як розподіленої системи. Які основні компоненти системи та яку роль вони виконують? Відповідай технічно, стисло, структуровано за компонентами.
     Відповідь Claude (скорочено). Discord використовує гібридну архітектуру, що поєднує Python-моноліт для бізнес-логіки та близько 20 мікросервісів на Elixir для обробки подій реального часу. Основні компоненти: Gateway – WebSocket-шлюз на Elixir/BEAM для підтримки мільйонів постійних з’єднань; REST API – Python-моноліт для операцій з акаунтами, серверами та каналами; Presence Service – окремий Elixir-сервіс відстеження онлайн-статусів; Rust Data Services – проміжний шар між бізнес-логікою та ScyllaDB; ScyllaDB – основне сховище повідомлень (C++, замінило Cassandra у 2022 році).
     Запит В.2 до Claude (Anthropic). Тема: шардинг та масштабованість Gateway.
     Поясни механізм шардингу WebSocket Gateway у Discord. Як обчислюється номер шарда? Чому це вирішує проблему горизонтального масштабування? Максимум 150 слів.
     Шардинг у Discord розподіляє WebSocket-з’єднання між незалежними вузлами. Номер шарда обчислюється за формулою: shard_id = (guild_id » 22) mod num_shards. Операція » 22 відкидає мілісекундну частину Snowflake ID, залишаючи стабільний ідентифікатор сервера. Це гарантує, що всі учасники одного guild потрапляють на один шард, що спрощує fan-out повідомлень – шард точно знає, яким сесіям розіслати подію, без міжшардової комунікації. Горизонтальне масштабування досягається збільшенням кількості шардів: нові вузли додаються без переписування логіки. Discord вимагає шардингу від ботів на 2500+ серверах, рекомендуючи не більше 1000 серверів на шард.
     Запит В.3 до Claude (Anthropic). Тема: порівняння ScyllaDB та Apache Cassandra.
     Порівняй ScyllaDB та Apache Cassandra з точки зору застосування у Discord. Чому Discord перейшов з Cassandra на ScyllaDB? Які конкретні метрики покращились після міграції?
     Відповідь Claude (скорочено). Apache Cassandra написана на Java, що означає залежність від JVM та паузи збирача сміття (GC pauses) – головне джерело непередбачуваних стрибків затримки у Discord. При 177 вузлах p99 затримка читання сягала 40–125 мс. ScyllaDB написана на C++ з архітектурою shard-per-core, що повністю усуває JVM GC. Після міграції: кількість вузлів скоротилась з 177 до 72 при тих самих обсягах даних; p99 затримка читання знизилась до стабільних 15 мс; p99 затримка запису стабілізувалась на рівні 5 мс. Додатково Discord розробив кастомну топологію на GCP: NVMe-диски у RAID0 для швидкості читання + Persistent Disk у RAID1 для надійності зберігання.
