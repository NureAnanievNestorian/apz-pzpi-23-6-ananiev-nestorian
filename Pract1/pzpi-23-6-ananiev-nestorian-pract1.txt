2 ЗАВДАННЯ
     Метою практичного заняття є ознайомлення з одним із шаблонів (патернів) проєктування програмного забезпечення з класичного каталогу «Банди чотирьох» (GoF), поглиблене вивчення його структури, призначення та практичного застосування, а також формування навичок проєктування слабозвʼязаних програмних систем із використанням перевірених архітектурних рішень.
     На основі проведеної роботи студент має підготувати презентацію, яка містить теоретичний вступ із класифікацією патернів GoF, детальний опис структури патерна, діаграму взаємодії обʼєктів, приклад реалізації з поясненням коду, порівняльний аналіз переваг і недоліків патерна, а також висновки щодо доцільності його застосування у реальних програмних проєктах.

3 ОПИС ВИКОНАНОЇ РОБОТИ
     3.1 Загальні відомості про шаблони (патерни) проєктування
     Шаблони (патерни) проєктування програмного забезпечення являють собою типові, перевірені практикою рішення для поширених задач, що виникають у процесі проєктування програмних систем. Вони описують узагальнений підхід до розвʼязання конкретної проблеми, не привʼязуючись до конкретної мови програмування або технологічного стеку, і служать спільним словником для розробників.
     Класичний каталог патернів «Банди чотирьох» (GoF – Gang of Four) визначає 23 патерни, які поділяються на три категорії. Породжувальні патерни (Creational) визначають механізми створення обʼєктів: Factory Method, Abstract Factory, Builder, Prototype, Singleton. Структурні патерни (Structural) описують способи організації класів та обʼєктів у більші структури: Adapter, Bridge, Composite, Decorator, Facade, Flyweight, Proxy. Поведінкові патерни (Behavioral) описують алгоритми та розподіл відповідальностей між обʼєктами: Chain of Responsibility, Command, Iterator, Mediator, Memento, Observer, State, Strategy, Template Method, Visitor.
     Патерн посередник (Mediator) належить до групи поведінкових патернів і спрямований на централізацію та спрощення взаємодії між обʼєктами у системі.
     3.2 Призначення патерна Mediator
     Патерн Посередник (Mediator) визначає обʼєкт-посередник, який інкапсулює логіку взаємодії між множиною обʼєктів. Завдяки цьому обʼєкти не взаємодіють один з одним напряму, а натомість спілкуються виключно через посередника. Ключова мета патерна – зменшення звʼязаності між компонентами системи, спрощення їх взаємодії та підвищення гнучкості архітектури.
     Передумовою для застосування патерна є ситуація, коли кількість прямих звʼязків між обʼєктами стає надмірно великою. Без посередника N обʼєктів утворюють N*(N−1)/2 звʼязків, тоді як за наявності посередника кількість звʼязків скорочується до N – кожен обʼєкт знає лише про посередника. Наочна аналогія з реального світу: авіадиспетчер координує всі переговори між літаками, і жоден пілот не звʼязується з іншим напряму.
     Проблеми, що виникають за відсутності посередника: висока звʼязаність – кожен обʼєкт безпосередньо залежить від деталей реалізації інших; складність підтримки – зміна одного обʼєкта вимагає каскадних змін у всіх повʼязаних; неможливість повторного використання – компоненти неможливо ізолювати та перевикористати в інших контекстах.
     3.3 Структура патерна Mediator
     Структура патерна Посередник складається з чотирьох ключових учасників. Mediator – інтерфейс, що визначає протокол взаємодії між колегами. ConcreteMediator – конкретна реалізація інтерфейсу Mediator, яка координує взаємодію між обʼєктами-колегами та зберігає посилання на всіх учасників взаємодії. Colleague – абстрактний базовий клас, що містить посилання на обʼєкт-посередник; є спільним предком для всіх обʼєктів-учасників. ConcreteColleague – конкретний учасник взаємодії, який надсилає запити посереднику та обробляє повідомлення, отримані від нього.
     Діаграма взаємодії між учасниками описує такий сценарій: ColleagueA ініціює подію шляхом виклику методу mediator.notify(this, "event"); ConcreteMediator отримує подію та визначає одержувача; медіатор виконує маршрутизацію та викликає метод colleagueB.react("event"); ColleagueB реагує на подію, не маючи жодних відомостей про ColleagueA.
     3.4 Реалізація патерна Mediator мовою Java
     Для демонстрації патерна Посередник реалізовано приклад взаємодії графічних компонентів – кнопки (Button) та текстового поля (TextBox) – через обʼєкт-посередник. Основні етапи реалізації: оголошення інтерфейсу Mediator з методом notify(); реалізація класу ConcreteMediator, що координує Button і TextBox; створення абстрактного класу Component із посиланням на mediator; реалізація конкретних компонентів Button і TextBox; демонстрація використання в методі main().

     3.5 Переваги та недоліки патерна Mediator
     До переваг патерна відносяться: слабка звʼязаність між компонентами – принцип єдиної відповідальності дотримано, оскільки кожен компонент відповідає лише за власну логіку; єдина точка контролю – вся логіка взаємодії зосереджена в одному класі-медіаторі, що спрощує її модифікацію; спрощення повторного використання – компоненти легко ізолювати та переносити до інших систем; легке розширення – нові колеги можна додавати без жодних змін у класах, що вже існують.
     До недоліків патерна відносяться: ризик утворення антипатерну «Бог-обʼєкт» (God Object) – ConcreteMediator може надмірно розростися та перевантажитися логікою, що порушує принцип єдиної відповідальності; концентрація складності – вся відповідальність за координацію зосереджується в одному класі; складність тестування – медіатор із великою кількістю колег важко тестувати ізольовано.


4 ВИСНОВКИ
     У результаті виконання практичного заняття було досліджено поведінковий патерн проєктування Посередник (Mediator) із класичного каталогу GoF. Проаналізовано його архітектурні особливості, структуру та сценарії взаємодії між учасниками – Mediator, ConcreteMediator, Colleague та ConcreteColleague.
     Практичне застосування патерна продемонстровано на прикладі взаємодії GUI-компонентів (Button і TextBox) мовою Java. Реалізація підтвердила, що патерн ефективно усуває пряму залежність між компонентами, зводячи кількість звʼязків із квадратичної до лінійної.
     Визначено основні переваги патерна: слабка звʼязаність, єдина точка контролю взаємодії та легкість розширення. Виявлено потенційний недолік – ризик виникнення антипатерну «Бог-обʼєкт» у ConcreteMediator за умови надмірного зростання логіки координації.
     Набуто практичних навичок у застосуванні патернів проєктування. Створено GitHub-репозиторій зі стандартизованою структурою, підготовлено слайди презентації та відеодоповідь. Практичне застосування патернів проєктування дозволяє підвищити повторне використання коду, масштабованість та підтримуваність програмних систем.

Лістинг В.1 – Реалізація патерна Посередник (Mediator) мовою Java:
 1  interface Mediator {
 2      void notify(Component sender, String event);
 3  }
 4  
 5  class ConcreteMediator implements Mediator {
 6      private Button button;
 7      private TextBox textBox;
 8  
 9      public ConcreteMediator(Button b, TextBox t) {
10          this.button  = b;
11          this.textBox = t;
12      }
13  
14      @Override
15      public void notify(Component sender, String event) {
16          if (sender == button && event.equals("click")) {
17              textBox.setText("Натиснуто кнопку");
18          }
19      }
20  }
21  
22  abstract class Component {
23      protected Mediator mediator;
24      public Component(Mediator m) { this.mediator = m; }
25  }
26  
27  class Button extends Component {
28      public Button(Mediator m) { super(m); }
29      public void click() {
30          mediator.notify(this, "click");
31      }
32  }
33  
34  class TextBox extends Component {
35      private String text = "";
36      public TextBox(Mediator m) { super(m); }
37      public void setText(String t) { this.text = t; }
38      public String getText()       { return this.text; }
39  }
40  
41  // Демонстрація використання
42  public class Main {
43      public static void main(String[] args) {
44          ConcreteMediator mediator = new ConcreteMediator(
45              new Button(null), new TextBox(null)
46          );
47          Button  btn = new Button(mediator);
48          TextBox box = new TextBox(mediator);
49          btn.click();
50      }
51  }
